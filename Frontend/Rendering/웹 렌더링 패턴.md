렌더링 방식 비교: CSR, SSR, SSG, ISR

1. CSR (Client-Side Rendering)
   서버: 최소한의 빈 HTML 파일과 JavaScript 번들 파일을 클라이언트에 전송.

클라이언트:

빈 HTML 표시 (흰 화면 단계).

JS 파일을 다운로드 및 실행.

JS 코드가 백엔드 API를 호출하여 데이터를 가져옴.

가져온 데이터를 기반으로 DOM 요소를 브라우저에서 실시간 생성/삽입하여 페이지를 완성.

특징: 초기 로딩은 느리지만, 한 번 로드되면 페이지 전환이 매우 빠름 (SPA).

2. SSR (Server-Side Rendering)
   서버: 매 요청마다(Request Time) 아래 과정을 수행.

필요한 데이터를 API/DB에서 가져옴.

데이터가 채워진 완성된 HTML을 즉석에서 생성.

완성된 HTML을 클라이언트에 전송.

클라이언트:

전달받은 완성된 HTML을 화면에 즉시 표시 (사용자가 내용을 바로 볼 수 있음).

JS 파일을 다운로드 및 실행하여 기존 HTML에 로직을 연결하는 Hydration 과정 수행.

Hydration 완료 후 클릭 등 상호작용 가능.

특징: SEO에 유리하고 초기 노출 속도가 빠르나, 서버 부하가 높고 페이지 이동 시마다 서버를 거쳐야 함.

3. SSG (Static Site Generation)
   서버: **빌드 시점(Build Time)**에 딱 한 번 실행.

데이터를 미리 가져와서 모든 페이지의 정적 HTML 파일을 생성.

생성된 HTML은 **CDN(Content Delivery Network)**을 통해 전송 준비 완료.

클라이언트:

미리 만들어진 HTML을 즉시 표시.

JS 다운로드 및 Hydration 수행.

특징: 가장 빠르고 서버 부하가 거의 없음. 하지만 데이터가 바뀌면 전체 서비스를 다시 빌드해야 함.

4. ISR (Incremental Static Regeneration)
   서버: 기본적으로 SSG와 동일하게 빌드 시점에 HTML을 생성하여 전송.

클라이언트: SSG와 동일한 과정(HTML 표시 → Hydration)을 거침.

서버의 추가 역할 (재검증 시): \* 설정된 revalidate 시간이 경과한 후 요청이 들어오면, 백그라운드에서 데이터를 다시 가져와 새로운 HTML을 생성하고 캐시를 업데이트함.

다음 방문자는 업데이트된 최신 페이지를 보게 됨.

특징: SSG의 속도와 SSR의 최신 데이터 유지 능력을 합친 방식.
