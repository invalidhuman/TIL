# Web Worker

## 이해를 위한 배경 지식

### 프로세스와 스레드

- **프로세스**: 실행 중인 프로그램 하나. 독립된 메모리 공간을 가짐
- **스레드**: 프로세스 안에서 실제로 코드를 실행하는 단위. 하나의 프로세스 안에 여러 스레드가 존재할 수 있고, 같은 메모리를 공유함
- Java, C++, Go 같은 언어는 스레드를 코드 안에서 여러 개 만들 수 있어서 하나의 프로세스 안에서 여러 코드가 동시에 실행될 수 있음

### 브라우저의 프로세스 구조 (Chrome 기준)

Chrome은 내부적으로 여러 프로세스로 나뉘어 있음

```
Chrome
├── Browser 프로세스      → 주소창, 탭 관리 등 브라우저 UI
├── Renderer 프로세스     → 탭마다 하나씩 존재. 실제 웹페이지를 처리
│   ├── 메인 스레드       → JS 실행 + 화면 렌더링 담당
│   ├── Compositor 스레드
│   └── 기타 스레드들
├── GPU 프로세스
└── Network 프로세스
```

탭 하나 = Renderer 프로세스 하나. 그 안에 여러 스레드가 있음

### JS 엔진 (V8)

- 브라우저는 JS를 직접 해석하지 않고, 내부에 **JS 엔진**을 내장해서 실행함
- Chrome의 JS 엔진 = **V8**
- **Node.js**도 V8을 꺼내서 서버 환경에 감싼 런타임. 브라우저 없이 서버에서 JS를 실행할 수 있는 이유

```
Chrome  → V8 내장 → 웹페이지의 JS 실행
Node.js → V8 내장 → 서버의 JS 실행
```

둘 다 같은 V8이지만, 주변에 제공되는 API가 다름

- 브라우저: `window`, `document`, `fetch`, Web API 등
- Node.js: `fs`, `http`, `os` 등 서버용 API

## JS는 싱글 스레드

JS 코드는 **메인 스레드 하나에서만 실행됨**. 동시에 두 줄이 실행되는 일은 없고, 항상 한 줄씩 순서대로 처리됨

Renderer 프로세스 안에 스레드가 여러 개 있어도, JS 실행은 그 중 **메인 스레드 하나만** 사용함. 나머지 스레드들은 렌더링 합성 등 다른 역할을 담당

그리고 메인 스레드는 JS 실행과 화면 렌더링을 **둘 다** 담당하기 때문에, JS가 오래 걸리면 렌더링이 밀림

```
정상 상태
[JS 실행] → [렌더링] → [JS 실행] → [렌더링] → ...

JS가 오래 걸릴 때
[JS 실행 중 ... 오래 걸림 ...] → (이 동안 렌더링 불가 → 화면 멈춤, 클릭 무반응)
```

이게 흔히 말하는 "버벅임"의 원인

### 비동기(async/await)와의 차이

비동기로는 이 문제를 해결하지 못하는 경우가 있음

|                      | 비동기 (fetch, setTimeout) | Web Worker            |
| -------------------- | -------------------------- | --------------------- |
| 스레드               | 메인 스레드 1개            | 스레드가 2개로 늘어남 |
| 원리                 | 기다리는 동안 다른 일 처리 | 진짜 동시에 실행      |
| CPU 연산이 무거울 때 | 해결 안 됨                 | 해결됨                |

`fetch`는 "네트워크 응답 기다리는 동안 다른 거 해"인데, 기다리는 동안 CPU를 안 쓰므로 괜찮음. 반면 이미지 처리처럼 CPU를 계속 써야 하는 작업은 `async/await`를 써도 메인 스레드가 막힘 → 이걸 Worker에 넘기는 것

## Web Worker란

브라우저가 Renderer 프로세스 안에 **별도의 Worker 스레드를 추가로 만들어주는** 것

```
Renderer 프로세스
├── 메인 스레드    → JS 실행 + 렌더링
└── Worker 스레드  → Web Worker의 JS 실행 (new Worker() 호출 시 생성)
```

무거운 연산은 Worker 스레드에서, 화면 그리기는 메인 스레드에서 **진짜로 동시에** 실행됨

### 생명주기

생명주기 = "언제 생기고 언제 사라지는가"

Web Worker는 **페이지(탭)에 종속**됨. 탭이 닫히면 Worker도 함께 사라짐

```
탭 열림 → new Worker() 호출 → Worker 실행 중 → 탭 닫힘 → Worker 종료
```

### 특징 및 제약

- DOM 접근 불가 (document, window 등 사용 못 함)
- 메인 스레드와 직접 메모리 공유 불가 → `postMessage`로 통신해야 함
- 네트워크 요청 인터셉트 불가
- 자동으로 생기지 않음. 개발자가 코드에서 직접 생성해야 함

## postMessage

Worker는 메인 스레드와 **완전히 분리된 실행 공간**이라 변수나 메모리를 직접 공유할 수 없음. 대신 메시지를 주고받는 방식으로 소통함

데이터를 넘길 때 직렬화(복사)가 일어남. 원본을 공유하는 게 아니라 복사본을 전달하는 것

```js
// main.js
const worker = new Worker("worker.js");
worker.postMessage({ data: largeArray }); // Worker에게 메시지 전송
worker.onmessage = (e) => console.log(e.data); // Worker의 응답 수신

// worker.js
self.onmessage = (e) => {
  const result = heavyComputation(e.data.data);
  self.postMessage(result); // 메인 스레드에게 결과 전송
};
```

흐름:

```
메인: "이 배열 정렬해줘" → postMessage(data)
Worker: 정렬 완료         → postMessage(result)
메인: "받았어"            → onmessage 실행
```

## 언제 쓰고, 언제 안 쓰나

### 쓸 때

- 이미지/영상 처리
- 대용량 CSV, JSON 파싱
- 암호화 연산
- 복잡한 수학 계산

### Trade-Off : 항상 쓰는 게 능사가 아닌 이유

**postMessage 비용**: 데이터를 주고받을 때 직렬화(복사)가 발생함. 작은 작업이면 오히려 이 비용이 더 클 수 있음

**코드 복잡도**: Worker 파일을 따로 만들고, 통신 구조를 따로 설계해야 함

기준: "이 작업이 메인 스레드를 100ms 이상 블로킹할 것 같다" 싶을 때 사용을 검토

### 일반 웹 서비스에서는?

API 호출, 리스트 렌더링, 폼 처리 등 대부분의 프론트엔드 작업은 CPU를 오래 쓰지 않아서 메인 스레드로 충분함. Figma 같은 복잡한 그래픽 에디터, 브라우저 기반 영상 편집 등 특수한 경우에 주로 사용

## React/Next.js에서 Web Worker

React, Next.js 같은 프레임워크를 써도 **Web Worker는 수동**임. 프레임워크가 자동으로 처리해주지 않음

프레임워크가 편하게 해주는 건 UI 상태 관리와 렌더링 패턴이고, "어느 스레드에서 실행할지"는 그 영역 밖

편하게 쓸 수 있게 해주는 라이브러리는 있음. `comlink` (Google)는 postMessage 통신을 함수 호출처럼 추상화해줌

```js
// worker.js
import { expose } from "comlink";
expose({ heavySort: (arr) => arr.sort() });

// main.js
import { wrap } from "comlink";
const worker = wrap(new Worker("./worker.js"));
const result = await worker.heavySort(bigArray); // postMessage 없이 함수처럼 호출
```
