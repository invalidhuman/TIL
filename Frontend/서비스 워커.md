# Service Worker

## 한 줄 정의

> 브라우저와 네트워크 사이에서 fetch 요청을 가로챌 수 있는 브라우저 레벨의 스크립트. 캐싱, 오프라인 지원, 푸시 알림에 쓰인다.

---

## 이해를 위한 배경 — TanStack Query 캐싱과의 차이

둘 다 "없으면 실제 요청, 있으면 저장된 거 반환"이라는 흐름은 같지만, 동작하는 레이어가 달라요.

```
TanStack Query:
요청 → 메모리에 없음 → fetch 실행 → 응답을 JS 메모리에 저장
같은 키로 재요청 → 메모리에 있음 → fetch 안 하고 메모리에서 반환

Service Worker:
요청 → Service Worker가 가로챔 → Cache Storage에 없음 → fetch 실행 → Cache Storage에 저장
같은 요청 재발생 → Service Worker가 가로챔 → Cache Storage에 있음 → fetch 안 하고 반환
```

핵심 차이는 **어디에 저장하냐**

```
[브라우저]
  └── Service Worker  ← 브라우저 레벨, 탭 밖에 존재
        └── [탭 / 페이지]
              └── [React 앱]
                    └── TanStack Query  ← JS 메모리 안에 존재
```

|                       | TanStack Query                 | Service Worker         |
| --------------------- | ------------------------------ | ---------------------- |
| 저장 위치             | JS 메모리 (휘발성)             | Cache Storage (영구적) |
| 새로고침하면          | 캐시 사라짐                    | 캐시 유지됨            |
| 탭 닫으면             | 캐시 사라짐                    | 캐시 유지됨            |
| 오프라인 지원         | 불가                           | 가능                   |
| HTML/CSS/JS 파일 캐싱 | 불가                           | 가능                   |
| 목적                  | 서버 상태 관리, 중복 요청 방지 | 네트워크 자체를 제어   |

TanStack Query는 "이 세션 동안 같은 API 중복 호출 방지"가 목적이고, Service Worker는 "브라우저 수준에서 네트워크 자체를 제어"하는 게 목적이에요.

또한 TanStack Query는 fetch가 나가기 전에 막는 게 아니라 응답 결과를 저장해두는 것. Service Worker는 fetch가 나가기 전에 브라우저 레벨에서 물리적으로 막을 수 있음

---

## 생명주기

Service Worker는 탭과 독립적으로 브라우저 레벨에서 생존함. 탭이 닫혀도 살아있고, 탭이 열리기 전부터 요청을 가로챌 수 있음.

```
등록 → 설치(install) → 활성화(activate) → 대기 (fetch/push 이벤트 수신)
                                              ↓
                                          탭이 닫혀도 유지
                                          브라우저 꺼지면 종료
```

Service Worker 업데이트 시 새 버전이 바로 적용되지 않고 기존 탭이 모두 닫힐 때까지 대기함. 이 때문에 사용자에게 "새 버전이 있습니다, 새로고침하세요"를 띄우는 UX 처리가 필요함

제약

- HTTPS 필수 (localhost 제외)
- DOM 접근 불가
- 페이지의 JS 변수에 직접 접근 불가

## 주요 역할

### 1. 캐싱 / 오프라인 지원

Service Worker가 fetch 이벤트를 가로채서 Cache Storage를 먼저 확인하고, 없으면 실제 서버로 보냄

```js
// service-worker.js
self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request).then((cached) => {
      return (
        cached ??
        fetch(event.request).then((response) => {
          // 응답을 Cache Storage에 저장
          const clone = response.clone();
          caches.open("v1").then((cache) => cache.put(event.request, clone));
          return response;
        })
      );
    })
  );
});
```

PWA(Progressive Web App)의 오프라인 지원이 이 원리로 동작함. 카카오맵, 트위터 라이트 등이 오프라인에서도 기본 화면이 뜨는 이유

### 2. 정적 리소스 캐싱

이미지, JS 번들, CSS 파일 같이 자주 안 바뀌는 파일을 Service Worker로 캐싱하면 두 번째 방문부터 네트워크 요청 없이 Cache Storage에서 반환. 로딩 속도 개선

### 3. 푸시 알림

탭이 닫혀 있어도 Service Worker는 살아있어서, 서버에서 푸시 메시지가 오면 Service Worker가 받아서 알림을 띄울 수 있음

푸시 알림 전체 흐름

```
1. 사용자가 알림 권한 허용
2. 브라우저가 FCM(Firebase Cloud Messaging) 등 푸시 서버에 구독 등록
   → 고유한 엔드포인트 URL 발급
3. 이 URL을 우리 서버에 저장
4. 알림 보낼 일이 생기면 → 우리 서버가 FCM에 요청
5. FCM이 브라우저로 푸시 메시지 전달
6. 브라우저가 Service Worker에게 'push' 이벤트 발생
7. Service Worker가 showNotification()으로 알림 띄움
```

```js
// service-worker.js
self.addEventListener("push", (event) => {
  const data = event.data.json();
  self.registration.showNotification(data.title, {
    body: data.body,
  });
});
```

웹에서 탭이 닫혀 있을 때 푸시 알림을 받으려면 Service Worker가 필수. 탭이 열려 있을 때만 알림을 받아도 된다면 WebSocket으로도 가능하긴 함

### 4. 백그라운드 Sync

오프라인 상태에서 발생한 요청(예: 폼 제출)을 저장해뒀다가, 네트워크가 복구되면 자동으로 재전송하는 기능

### 5. MSW (Mock Service Worker)

테스트/개발 환경에서 실제 API 서버 없이 mock 응답을 돌려줌. Service Worker가 fetch를 가로채서 미리 정의한 응답을 반환하는 원리

## 이벤트 리스닝

Service Worker는 특정 이벤트가 발생했을 때 반응하는 방식으로 동작함. `addEventListener`로 이벤트를 등록하는 것

"구독"이라는 개념과 유사하지만, WebSocket과는 다름

|                                | Service Worker 이벤트 리스닝                                  | WebSocket                           |
| ------------------------------ | ------------------------------------------------------------- | ----------------------------------- |
| 연결 방식                      | 브라우저가 FCM과 연결 관리, 이벤트 오면 Service Worker를 깨움 | 클라이언트-서버 간 연결을 직접 유지 |
| Service Worker/클라이언트 상태 | 평소엔 잠들어 있음                                            | 연결 유지 중엔 항상 깨어 있음       |
| 방향                           | 서버 → 클라이언트 (단방향)                                    | 양방향                              |

공통점은 "어떤 일이 생기면 알려줘"라는 구조이고, 차이는 연결을 누가 유지하냐임

## 프록시 vs 리버스 프록시 — Service Worker와의 관계

### 프록시(Forward Proxy)

클라이언트 앞에 서는 대리인. 클라이언트가 설정함

```
클라이언트 → [프록시] → 서버
```

서버는 클라이언트의 실제 IP를 모름. 용도: VPN, 익명성 확보, 사내 네트워크 필터링

### 리버스 프록시

서버 앞에 서는 대리인. 서버 운영자가 설정함

```
클라이언트 → [리버스 프록시] → 서버
```

클라이언트는 리버스 프록시만 알고, 실제 서버의 IP/구조를 모름. 용도: 로드 밸런싱, SSL 처리, 서버 보호. Nginx가 대표적

물리적 위치는 둘 다 중간이지만, 누가 설정하고 누구를 위한 것이냐가 다름

|             | Forward Proxy   | Reverse Proxy  |
| ----------- | --------------- | -------------- |
| 설정 주체   | 클라이언트      | 서버 운영자    |
| 누구를 위함 | 클라이언트      | 서버           |
| 숨기는 것   | 클라이언트 신원 | 서버 위치/구조 |

### Service Worker와 리버스 프록시 비교

Service Worker를 "브라우저 안의 리버스 프록시"라고 비유하는 이유는 흐름이 유사하기 때문

```
공통점: 중간에서 요청을 받아 → 캐시 확인 → 없으면 원래 목적지로 전달
```

하지만 목적까지 같은 건 아님

|                  | Nginx (리버스 프록시)  | Service Worker               |
| ---------------- | ---------------------- | ---------------------------- |
| 동작 레벨        | 네트워크 레벨          | 브라우저 레벨                |
| 설정 주체        | 서버 운영자            | 프론트엔드 개발자            |
| 서버 위치 숨기기 | 가능                   | 불가능 (URL이 코드에 노출됨) |
| 주목적           | 서버 보호, 로드 밸런싱 | 캐싱, 오프라인, 푸시 알림    |

Service Worker는 브라우저 안에 있기 때문에 실제 서버의 URL은 코드에 그대로 노출됨. 서버를 숨기는 건 Service Worker의 역할이 아님. 리버스 프록시 비유는 "흐름의 구조가 비슷하다"는 선에서만 유효함

## 웹 vs 앱에서의 Service Worker 역할 비교

| 역할              | Service Worker (웹)               | React Native / Expo                             |
| ----------------- | --------------------------------- | ----------------------------------------------- |
| 캐싱              | Cache Storage API                 | AsyncStorage, MMKV, SQLite                      |
| 오프라인 지원     | Service Worker가 캐시에서 반환    | NetInfo + 로컬 DB                               |
| 푸시 알림         | Service Worker가 push 이벤트 수신 | APNs (iOS), FCM (Android), `expo-notifications` |
| 백그라운드 sync   | Background Sync API               | `expo-background-fetch`, `expo-task-manager`    |
| 네트워크 인터셉트 | fetch 이벤트로 가로챔             | 없음 (axios interceptor는 앱 레벨)              |

Service Worker는 웹 브라우저 전용 기술. React Native 앱에서는 OS 레벨의 기능들이 각 역할을 대신함

## React/Next.js에서 Service Worker

Service Worker는 React/Next.js가 자동으로 처리해주지 않음. 프레임워크가 편하게 해주는 건 UI 상태 관리와 렌더링 패턴이고, 브라우저 레벨의 Service Worker는 그 영역 밖

Next.js에서는 `next-pwa` 라이브러리로 Service Worker 설정을 자동화할 수 있음

Service Worker 적용 시 신경 써야 할 점

- 캐시 전략: 언제 캐시를 갱신할지 (stale-while-revalidate, cache-first 등)
- Service Worker 업데이트 시 사용자에게 "새 버전이 있습니다" 알림 UX 처리
- HTTPS 환경 필수
